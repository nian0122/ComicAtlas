# ComicAtlas 0.2 产品定位与原则

**版本**: 0.2  
**日期**: 2026-07-16  
**状态**: Canonical

---

## 一句话定位

> **ComicAtlas 是一个本地部署的个人漫画仓库（Personal Comic Library）。**

---

## 核心能力

```
ComicAtlas
├── 阅读（Reading）
└── 管理（Management）
```

只有这两个核心能力。不做社区、不做推荐、不做排行榜、不做收藏、不做账号、不做权限、不做管理员后台。

原因：**用户就是管理员。**

---

## 核心原则

### 原则一：阅读是唯一核心体验

> **管理服务于阅读，阅读是整个产品唯一的核心体验。**

这条原则影响所有设计决策：

- 首页不是 Dashboard，而是阅读首页。
- 漫画详情页没有任何管理按钮。
- 管理功能使用独立的 Layout 和入口。
- API 优先满足 Reader 的消费场景。

### 原则二：阅读和管理彻底分离

阅读是高频每日路径，管理是低频维护路径。两者在 UI、路由、Layout、API 职责上都应清晰分离。

### 原则三：个人项目导向

不引入面向多用户平台的设计：

- 无用户系统、无角色权限。
- 无推荐算法、无排行榜、无收藏。
- 无运营统计（PV/UV/在线人数）。

### 原则四：保留稳定基础设施

0.2 重构的是产品层和前端架构，不是数据库、Worker、MQ、导入流程、存储模型、阅读器等已经验证的基础设施。

### 原则五：导航按用户任务设计

管理侧导航表达用户任务（漫画、导入、存储、元数据、设置），而不是技术模块（Cover、Scan、Recovery 等）。

### 原则六：迁移优先，功能后置

0.2 前半程只做架构、路由、Layout、页面归属调整；Category、封面管理等新功能在架构稳定后再叠加。

### 原则七：每个阶段都可运行

每个迁移阶段结束时，项目必须可启动、可测试、Playwright 通过，不存在半废弃状态。

## 范围边界

### 0.2 包含

- 阅读 / 管理双 Layout 与双路由树。
- 阅读首页、漫画库、详情、阅读器、历史。
- 管理：漫画管理、导入、存储、元数据（Category + Tag）、设置。
- 前端目录结构重组。
- API 职责按场景梳理（Controller 保留，DTO 按场景拆分）。
- 新增 `category` 表，与 `tag` 分离。

### 0.2 不包含

- 新来源适配（EHentai、SMB 等）。
- 封面裁切、批量编辑、评分、笔记。
- 多用户、权限、账号系统。
- 推荐、收藏、排行榜、社区。
- 运营类 Dashboard 和统计。

## 术语

| 术语 | 含义 |
|------|------|
| Reading | 阅读域，每天使用的核心路径 |
| Management | 管理域，仓库维护操作 |
| Layout | Vue Router 的顶层布局组件 |
| Comic | 漫画聚合根 |
| Category | 作品类型，一级单选 |
| Tag | 内容标签，多选 |
| Catalog | 漫画内部目录树（卷 / 部 / 章节） |
| Storage | 文件存储管理，包括 HQ / LQ / Thumbs |
