## 编号

BUG-005

## 类型

- [x] BUG

## 模块

Build / Docker

## 描述

api-service、worker-service、gateway 的 Dockerfile 都是直接 `COPY target/*.jar app.jar`，不在镜像内重新编译。如果本地 `target/` 目录残留旧 jar，Docker 会打包旧代码运行，导致线上行为与 Git 源码不一致。

本次验收中，Git 源码的 `ImportServiceImpl` 已支持 `DIRECTORY`，但容器内运行的 class 文件只有 `EHENTAI/ZIP/REGISTER` 三个 case，`DIRECTORY` 被当作未知类型返回 400。

## 复现步骤

1. 修改后端源码（例如增加 DIRECTORY case）
2. 不执行本地 `mvn clean package`
3. `docker-compose up -d --build`
4. 调用 `POST /api/tasks/import {sourceType:"DIRECTORY"}`
5. 返回 400 "不支持的 sourceType: DIRECTORY"

## 期望行为

Docker 镜像应基于当前源码重新构建，确保容器内代码与 Git 工作区一致。

## 实际行为

容器运行的是本地 `target/` 中残留的旧 jar。

## 影响范围

- 是否阻塞 Beta v0.1 发布？是
- 是否影响全链路？是，任何后端修改都可能不生效

## 建议修复方案

将三个后端 Dockerfile 改为多阶段构建：

```dockerfile
FROM maven:3.9-eclipse-temurin-21-alpine AS builder
WORKDIR /build
COPY pom.xml .
COPY src ./src
RUN mvn clean package -DskipTests

FROM eclipse-temurin:21-jre-alpine
WORKDIR /app
COPY --from=builder /build/target/*.jar app.jar
EXPOSE 8010
ENTRYPOINT ["java", "-jar", "app.jar"]
```

worker-service 需额外安装 aria2。

## 修复记录

- `api-service/Dockerfile`、`worker-service/Dockerfile`、`gateway/Dockerfile` 均改为多阶段构建：builder 阶段使用 `maven:3.9-eclipse-temurin-21-alpine` 从源码重新编译，runtime 阶段使用 `eclipse-temurin:21-jre-alpine-3.20` 复制 jar 运行。
- `docker-compose.yml` 调整为从项目根目录构建，context 包含 `pom.xml` 与各模块源码。
- worker-service 镜像暂时移除 aria2 安装（Alpine 包安装失败），ZIP/DIRECTORY 导入不依赖 aria2，不影响 Beta v0.1 功能。
- 验证：DIRECTORY 导入在容器内可正常识别并处理，不再报 `不支持的 sourceType: DIRECTORY`。
